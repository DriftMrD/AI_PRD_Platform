# 对话提炼 SKILL（dialogue）

**适用输入**：用户未上传附件，仅提供对话式文本（单轮描述、多轮补充、口语化需求碎片）。目标是**从对话中归纳**结构化 PRD，对齐 `prd-template.md`。

## 核心原则

1. **按时间线阅读**：若有「第 N 轮需求」或多段对话，按顺序理解；**后文对前文的修正/补充优先于早期表述**。
2. **归纳不臆造**：只把对话中明确说过的需求写入 PRD；未提及的字段留空或标注 **`> 【ai_inferred】`** 并说明推断依据。
3. **禁止编造决策**：PM 姓名、AR ID、排期、API 版本号等对话未给出的信息——勿捏造。
4. **口语 → 结构化**：把「我想要…」「最好还能…」等口语，改写为清晰的用户故事、范围条目与验收标准，但**不改变原意**。
5. **冲突处理**：多轮需求若矛盾，采用**最新一轮**的说法，或在 PRD 中用 **【ai_inferred】** 注明「前文 A，后文改为 B」。

## 对话内容 → 模板章节

| 对话中常见表达 | 目标章节 |
|---------------|---------|
| 背景、痛点、为什么要做 | §1 |
| 要做/不做、本期/以后 | §2 |
| 用户怎么用、场景、流程 | §3 |
| 界面、按钮、页面长什么样 | §4 |
| 怎样算做完、验收标准 | §5 |
| 性能、兼容、权限、依赖 | §6 |
| 担心什么、已知限制 | §7 |

## 执行步骤

1. 通读全部对话文本（含「当前 PRD 草稿」段落，若有则在其基础上修改）。
2. 提取：功能名、目标用户、核心流程、边界与不做项。
3. 拟定 **`# <feature_name> PRD`**：功能名从对话推断，简洁可读。
4. 信息不足处用 **【ai_inferred】** 标注，勿静默补全。
5. **§3** 至少 3 条用户故事；对话里每个独立场景尽量对应一条 Story。
6. **§5** 验收项从对话中的「必须」「应该」提炼为 `- [ ]` 列表。

## 质量要求

- §3 用户故事至少 3 条，每条至少 1 主流程 + 1 边界/异常场景
- §4 对话未给 Figma 时，节点 ID 留空并 **【ai_inferred】** 注明「待补充 Figma」
- §5 验收项用 `- [ ]` checkbox 列表，可测试、可勾选
