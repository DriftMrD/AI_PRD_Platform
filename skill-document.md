# 文档转换 SKILL（document）

**适用输入**：用户上传了附件（PRD 草稿、需求文档、大纲、混杂的 UI 说明等）。支持格式：`.md` `.txt` `.docx` `.pdf`。目标是**保真重组**已有文档，对齐 `prd-template.md` 结构。

## 附件与补充说明（同时存在时）

用户可能**既上传附件，又在「用户原始需求」里打字补充**。处理方式：

| 输入块 | 角色 |
|--------|------|
| 【附件 1…N】 | **主原料**：保真搬运文档正文 |
| 【补充说明】 | **修改 / 增量指令**：如「重点改 §3」「新增导出功能」「删掉 XX」 |

规则：

1. 先通读全部附件建立基线 PRD，再按用户补充说明做定向修改或增量。
2. **冲突时用户补充优先**：附件写 A、用户说改成 B → 输出 B，可用 **【ai_inferred】** 注明「按用户补充覆盖附件原表述」。
3. 用户补充仅涉及局部时，**未提及章节保持附件原意**，勿整篇重写。
4. 用户补充是全新需求、附件无相关内容时，将补充写入合适章节并 **【ai_inferred】** 标注来源。

## 核心原则

1. **保真搬运**：附件中的规则、数字、枚举、SDK 字段名、埋点名等必须保留，不改写含义（除非用户补充明确要求修改）。
2. **推断须标注**：模板里没有而必须从上下文补全的内容，用 **`> 【ai_inferred】`** 标注原因。
3. **禁止编造决策**：商业优先级、明确日期、未给出的 API/版本号——勿捏造；用占位符或「待 PM 填写」。
4. **用户故事与场景**：若原文是「场景 A/B/C」或交互说明，映射到 **§3**；每个 Story 至少包含主流程 + 一条边界或异常 Scenario，使用 **GIVEN / WHEN / THEN / AND**。
5. **范围**：原文中的「不做」「后续」「本期」归类到 **§2.2 / §2.3**；本期交付列入 **§2.1**。

## 章节映射（常见来源 → 模板位置）

| 原文常见块 | 目标章节 |
|-----------|---------|
| 版本、负责人、链接、AR | 元信息表 |
| 背景、目标、Why、痛点 | §1（1.1～1.3） |
| In/Out scope、本期/下期 | §2 |
| 用户故事、场景、用例、流程 | §3 |
| UI、Figma、控件、交互 | §4 |
| 验收、Definition of Done | §5 |
| SDK、依赖、兼容、性能、网络、隐私、权限 | §6 |
| 风险、已知问题 | §7 |
| 竞品、wiki、历史文档 | §8 |
| （新建转换） | 变更历史：一行初版 |

## 执行步骤

1. 通读【补充说明】（若有）与全部附件，列出事实清单，明确哪些是基线、哪些是修改指令。
2. 拟定 **`# <feature_name> PRD`** 标题：优先取自附件标题或用户补充中的功能名。
3. 按模板顺序填空：**元信息**能填则填；缺失项留空或写 `-`，勿猜 PM 姓名。
4. **§3** 优先：把分散的「界面/场景」收成 Story；Scenario 用 `Scenario 3.x.y` 连续编号。
5. **§4**：若原文只有「图 1、图 2」而无 Figma 链接，节点 ID 可空，**【ai_inferred】** 注明「待补充 Figma」。
6. **§5**：验收项从原文「规则」「必须」「不应」转化为 `- [ ]` 勾选列表，尽量可验收、可测试。
7. 全文检查：分隔线、表格对齐、列表层级与模板一致。

## 质量要求

- §3 用户故事至少 3 条，每条至少 1 主流程 + 1 边界/异常场景
- §4 UI 设计若原文无 Figma 链接，节点 ID 留空并标注「待补充 Figma」
- §5 验收项用 `- [ ]` checkbox 列表，可测试、可勾选
